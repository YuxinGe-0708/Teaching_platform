## 2 系统需求概述

### 2.1 需求规定

**输入项目**：
- **用户身份信息**：账号密码登录凭证、角色（学生/教师/管理员）及权限标识。
- **课程数据**：课程名称、简介、封面、教学班邀请码、教学计划配置。
- **作业提交**：
  - 客观题：选择题选项、判断题答案、填空题文本。
  - 主观题：文本输入、拍照或文件上传。
  - 编程题：通过 WebIDE完成的代码内容及编程语言选择。
- **AI 提问**：学生在视频播放或作业场景中输入的文本提问及课程上下文信息。

**输出项目**：
- **平台首页**：根据未登录/已登录状态动态渲染教学空间、资源广场入口及通知。
- **课程列表**：学生端展示已选课程与进度，教师端展示已创建课程与管理入口。
- **作业批改结果**：
  - 客观题：系统自动对比标准答案并即时出分。
  - 编程题：调用第三方云端判题 API，返回 AC/WA/TLE/CE 等标准化判题结果与解析。
  - 主观题：展示教师的分数、评语或批改附件。
- **成绩数据面板**：包含作业/考试加权平均分、全班分数段直方图、及格率等可视化看板。
- **AI 答复**：知识点闪退卡片、思维导图、结构化笔记总结。

**功能性能要求**：
- **响应速度**：前端界面操作响应时间应在用户可接受范围内，非外网依赖的查询接口应快速响应。
- **评测并发**：利用 Redis 异步消息队列削峰，防止考试截止前的高并发提交导致系统宕机。
- **稳定性**：系统部署于单台云服务器，在小中规模并发下评测系统无宕机，核心业务接口测试全覆盖。

### 2.2 业务目标

本项目旨在建设一个集**在线教学管理**、**作业全流程处理**、**代码实训评测**及 **AI 辅助学习**于一体的一体化高校教学实训平台。

- **教学空间管理**：支持教师创建课程与班级、上传资源、配置教学计划；支持学生选课、退课与学习进度跟踪。
- **作业全流程处理**：覆盖客观题自动批改、主观题人工批改与编程题云端评测，支持成绩统计与导出，形成教学闭环。
- **代码实训评测**：对接第三方公有云判题 API，舍弃高风险的本地沙盒自建，实现零运维负担的多语言在线判题。
- **AI 辅助学习**：引入大语言模型，在课件观看、代码编写与日常复习环节提供实时答疑、知识点总结与思维导图生成。

### 2.3 运行环境

#### 2.3.1 设备
- **计算机设备**：用户可通过台式机或笔记本电脑流畅访问系统网页端，推荐支持 Chrome 或 Edge 等现代浏览器的设备。

#### 2.3.2 支持软件
- **操作系统**：开发及客户端兼容 Windows、MacOS 及 Linux 的主流操作系统。
- **数据库与中间件**：采用 MySQL 8.0 作为持久化数据存储，引入Redis实现热点数据缓存、会话管理与评测队列削峰。
- **编程语言和框架**：
  - 后端：Java 8/11，基于 Spring Boot 2.7/3.0 构建，Maven 进行依赖管理。
  - 前端：采用 Vue 3 构建工程化界面，集成 Monaco Editor 提供代码编辑能力。

### 2.4 设计约束

系统整体设计应满足课程大作业要求，并符合企业级应用的基本规范：

- **角色隔离与权限约束**：
  - 严格隔离学生、教师、管理员三种角色的操作权限。学生仅可访问个人选课及成绩数据，教师仅可管理自身创建的课程，管理员负责全局配置与日志监控。
- **架构分层约束**：
  - 严格遵循“控制层-业务逻辑层-数据访问层”的单一职责分层架构。前端采用组件化开发，后端统一封装 `Result<T>` 响应体。模块间通过服务接口解耦，不得直接操作对方数据表。
- **外部服务依赖约束**：
  - 评测核心逻辑依赖第三方云 API，系统需设计网络超时重试机制与降级处理策略，确保外部服务异常时系统主流程不崩溃。

## 3 系统接口设计
### 3.1 用户接口
系统将提供针对性用户页面（UI）：
- **平台公共接口**：平台主页（未登录状态）。
- **学生操作面板**：包含我的课程、班级管理、作业平台、智能答疑区等视图控制。
- **教师操作面板**：批改作业面板、新增课程弹窗、成绩生成视图等视图控制。
- **管理员面板**：系统配置与用户状态列表视图。

### 3.2 与其他软件、硬件接口
- **外部在线评测接口(Judge API)**：采用HTTP/HTTPS接口，系统发送JSON格式（包含代码内容、语言限制参数等），异步获取并展示运行结果（如AC、WA等）。
- **大模型AI接口(AI BaaS)**：采用HTTP/HTTPS发送文本提问及课程上下文至AI服务商，返回知识总结等自然语言结果反馈给学生。
- **前后台HTTP/RESTful接口**：使用标准的JSON交互模式实现前端与后端的统一通讯。

### 3.3 系统数据结构设计
#### 3.3.1 逻辑结构设计要点
系统核心数据实体如下：
- **用户实体 (User)**：userID, username, password, role, contact (用于存储基本身份和角色体系)。
- **课程实体 (Course)**：courseID, courseName, teacherID, description, status。
- **作业与实训实体 (Assignment/Task)**：assignmentID, courseID, title, type, dueDate, content。
- **评测提交实体 (Submission)**：submissionID, assignmentID, studentID, code/content, status, score。
- **成绩归档实体 (ScoreTranscript)**：transcriptID, studentID, courseID, finalScore。

#### 3.3.2 物理结构设计要点
- 上述概念将映射到MySQL中的持久化表 (`Users`, `Courses`, `Assignments`, `Submissions`, `Scores`等)。
- 使用自增主键或雪花ID保障记录唯一性。
- 通过外键（如courseID, studentID）约束建立跨表联系，但应用内解耦业务避免大量外联查询。

#### 3.3.3数据结构与程序关系
系统后端采用`Controller - Service - Repository`逐层调用处理结构。例如，课程页面通过调用`CourseService`以获取当前教师ID作为输入过滤`Courses`表，再封装为视图返回给前端面板。

## 4 系统数据库设计

### 4.1 数据库选择

系统采用 MySQL 作为核心关系型数据库，用于持久化存储用户、课程、班级、作业、考试、提交记录、成绩、通知、日志、评测任务、AI 会话等结构化业务数据。

选择 MySQL 的原因如下：

1. 支持事务处理，适合选课、提交、批改、成绩更新等需要一致性的业务场景。
2. 支持多表关联查询，便于实现课程、学生、教师、作业和成绩之间的数据关联。
3. 与 Java 8、Spring Boot、MyBatis 技术栈兼容性较好。
4. 部署简单、维护成本低，适合课程项目和中小规模教学平台使用。
5. 后续可通过读写分离、分库分表等方式扩展系统能力。

系统同时引入 Redis 作为辅助缓存组件，主要用于：

1. 缓存首页公告、课程列表、题目详情等高频访问数据。
2. 缓存用户登录状态、验证码、权限信息等短期数据。
3. 支撑作业提交、评测状态查询等高峰场景下的削峰和限流。
4. 临时保存第三方评测或 AI 调用过程中的中间状态。

数据库设计遵循以下原则：

1. 核心业务数据必须持久化保存。
2. 高频访问数据可进行缓存优化。
3. 重要状态字段应统一枚举值。
4. 关键业务表应保留创建时间、更新时间和逻辑删除字段。
5. 涉及用户、课程、成绩等敏感数据时，必须保证权限边界清晰。

### 4.3 数据库关系设计

系统主要数据关系如下：

1. 用户与学生信息、教师信息为一对一关系。
2. 教师与课程为一对多关系。
3. 课程与班级为一对多关系。
4. 学生与课程为多对多关系，通过选课记录表维护。
5. 课程与教学资源为一对多关系。
6. 课程与作业/考试任务为一对多关系。
7. 学生与提交记录为一对多关系。
8. 任务与提交记录为一对多关系。
9. 学生与成绩为一对多关系。
10. 通知公告通过 target_type 和 target_id 支持全体、角色、课程、班级、个人等多种推送范围。
11. 外部服务记录表与提交记录、任务、用户进行弱关联，用于保存评测和 AI 调用历史。

### 4.5 数据库安全性考虑

1. 用户密码采用 BCrypt 等不可逆方式加密存储。
2. 学生仅可访问自己的课程、作业、考试、成绩和通知数据。
3. 教师仅可访问自己创建课程下的班级、资源、提交和成绩数据。
4. 管理员可管理用户、公告、配置和日志，但关键操作需要记录。
5. 数据库账号遵循最小权限原则。
6. 禁止在代码仓库中保存数据库密码、API Key 等敏感信息。
7. 外部评测和 AI 调用结果不得保存敏感隐私内容。
8. 所有用户输入参数必须进行校验，防止 SQL 注入。
9. 重要业务数据应定期备份，包括用户、课程、提交记录和成绩数据。
10. 对课程、任务、资源等重要业务数据建议采用逻辑删除方式，便于恢复和审计。

## 5 运行设计
### 5.1运行模块组合
主要的运行模块切片分为：系统验证与授权拦截模块、基础教学空间路由模块、核心业务CRUD服务模块，以及对外的第三方服务调度池模块（评测防抖/调用重试请求器）。

### 5.2运行控制
- 生命周期控制：评测任务采用“先入库为Pending、发至评测机后置为Judging、完成更新为具体状态”的三段式状态机流转控制。对于外部接口设置超时(timeout)及最大自动重试(retry)策略。

### 5.3运行时间
针对教务及一般检索服务，服务器业务逻辑交互处理时间应在合理毫秒级别（通常<500ms），评测请求因为资源隔离消耗，支持异步回调以便快速释放主线程，不使客户端阻塞失效。
